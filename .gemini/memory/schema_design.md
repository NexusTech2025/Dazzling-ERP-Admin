# Schema Design Standard (V3.0)

> [!IMPORTANT]
> The database schema is defined in decoupled modular JSON files located in [Config/Schema/](file:///E:/NAST/Dazzling/GAS/DazzlingDB/Config/Schema/).
> All other schema `json` files from `src/Schema` are outdated and **MUST** be ignored. The isolated JSON files inside `DazzlingDB/Config/Schema/` grouped by domain are the final and single source of truth for all database tables and fields.

---

## 1. File Naming Convention

- Files must be named after the entity they represent.
- Use **PascalCase** for the filename (e.g., `CourseType.json`, `StudentFeeAccount.json`).
- Extension must be `.json`.

---

## 2. Identifier Standardization

As of V2.1, the project has standardized on the following naming convention for identifiers to align the frontend state with the database schema:

- **Primary Keys**: Always use `[entity]_id` (e.g., `course_id`, `batch_id`, `teacher_id`).
- **Legacy Note**: References like `item_id` (used for Courses in some places) are deprecated and MUST be mapped to `course_id` in Data Mappers.
- **Foreign Keys**: Always mirror the target's primary key name (e.g., a Batch table linking to a Teacher should use `teacher_id`).

---

## 3. Root Object Structure

Decoupled schema JSON files are structured as a single root object with the following properties:

| Property | Type | Required | Description |
| :--- | :--- | :--- | :--- |
| `primaryKey` | `string` | Yes | The name of the primary key field for the entity. |
| `columns` | `object` | Yes | Key-value pairs mapping column names to their property definitions. |
| `relations` | `object` | No | Key-value pairs mapping relationship names to their definitions. |

---

## 4. Column Properties Structure

Each key under the `columns` object represents a field in snake_case and supports the following properties:

| Property | Type | Description |
| :--- | :--- | :--- |
| `type` | `string` | Data type: `string`, `number`, `json`, `auto`, `boolean`, `date`, `datetime`. |
| `required` | `boolean` | If `true`, the field must have a value on write. |
| `unique` | `boolean` | If `true`, the value must be unique across all records in the table. |
| `choices` | `array` | Required if the field represents an enum/set. Lists allowed string options. |
| `maxLength` | `integer`| The maximum string length allowed (typically `255`). |
| `default` | `any` | The default value if none is explicitly provided. |
| `idPrefix` | `string` | Required if `type` is `auto`. The prefix used for autogenerated primary keys (e.g., `CRS`, `STU`). |
| `editable` | `boolean` | If `false`, this field is read-only (e.g., autogenerated IDs). |

---

## 5. Relation Properties Structure

Each key under the `relations` object represents a relationship name and supports the following properties:

| Property | Type | Description |
| :--- | :--- | :--- |
| `type` | `string` | The relation cardinality: `belongsTo`, `hasOne`, `hasMany`, `belongsToMany`. |
| `target` | `string` | The name of the target entity in PascalCase (e.g., `CourseType`, `Student`). |
| `foreignKey` | `string` | The column name in the source or target entity establishing the link. |

---

## 6. Standard Data Types & Conventions

- **IDs**: Use type `auto` for primary identifiers with a suitable `idPrefix` (e.g., `STU` for Student).
- **Enums**: Use type `string` coupled with a `choices` array listing the exact lowercase or standard case options.
- **Timestamps**: Use `datetime` for `created_at` or `updated_at`.
- **Dates**: Use `date` for birthdays (`dob`) or schedule dates without time.
- **Metadata/Complex Data**: Use `json` for unstructured or semi-structured data.

---

## 7. Cohesive Schema Groupings

To maintain system integrity and ease of navigation, schemas are decoupled into individual JSON files and grouped into six functional domain subdirectories under `E:\NAST\Dazzling\GAS\DazzlingDB\Config\Schema/`:

### Domain A: Students (`Config/Schema/Students/`)
*Entities centered around the student profile, background, and identity.*
- `Student.json` (Core identity and status)
- `Address.json` (Residency details linked to student_id)
- `ContactInfo.json` (Communication and emergency details)
- `Education.json` (Prior qualifications and academic background)
- `StudentLead.json` (Unregistered leads and sales pipeline prospects)

### Domain B: Academic (`Config/Schema/Academic/`)
*The fundamental building blocks of the academic structures and schedules.*
- `CourseType.json` (High-level segments or categories of courses)
- `Course.json` (Individual subjects or academic programs)
- `Batch.json` (Operational instances of courses with schedule/teacher)
- `BatchAllocation.json` (Student enrollment mapped to batches)
- `Enrollment.json` (Links a student to a specific course/package program)
- `Package.json` (Definitions for bundled course offerings)
- `PackageItem.json` (Individual items/courses within a package bundle)
- `PackagePerk.json` (Additional benefits included in packages)

### Domain C: Staff (`Config/Schema/Staff/`)
*Entities representing the institute's workforce and resource management.*
- `Teacher.json` (Faculty profiles and professional assignments)
- `TeacherSubject.json` (Maps teachers to courses they are certified to teach)
- `TeacherAttendance.json` (Staff attendance logs)
- `TeacherDocument.json` (Staff uploaded identification/qualifications)
- `TeacherSalaryConfig.json` (Specific teacher salary config structures)
- `TeacherPaymentTransaction.json` (Detailed teacher payroll transactions)

### Domain D: Finance (`Config/Schema/Finance/`)
*All entities managing pricing, plans, billing, and transactional history.*
- `FeePlan.json` (Master pricing templates and financial setups)
- `StudentFeeAccount.json` (Tracks the overall student financial state)
- `Installment.json` (Scheduled payment breakdowns)
- `Payment.json` (Individual receipt/transaction records)
- `FeeAdjustment.json` (Modifications to base fees or manual discounts)

### Domain E: Core (`Config/Schema/Core/`)
*System-wide shared entities.*
- `Branch.json` (Physical or logical institute centers)
- `PromoCode.json` (Discount and marketing codes)

### Domain F: Auth (`Config/Schema/Auth/`)
*System security and sessions.*
- `User.json` (Credentials and login configuration)
- `Session.json` (Active login tokens)

## 10. Strict Field Alignment (UI-to-Schema)

To ensure zero-mapping overhead and data integrity, the frontend `formData` keys **MUST** perfectly match the field names defined in the schema.

### Example (Student Entity):
- **❌ Legacy UI Name**: `fullName`, `mobile`, `dob`
- **✅ Schema Correct Name**: `student_name`, `phone`, `dob` (matches keys in `Student.json`)

> **Rationale**: Casing and names must be exact drops to allow direct API transmission.

---

## 11. Cross-Entity Identifier Resolution

When an entity references another, the UI **MUST** store and transmit the target's **Primary Key** (e.g., `course_id`), rather than its semantic name.

### Standard for Many-to-Many Assignment (e.g., Teacher Subjects):
1. **Frontend State**: Store as an array of IDs: `subjects: ['CRS-001', 'CRS-002']`.
2. **Persistence (API Payload)**: Save as raw JSON array `['CRS-001', 'CRS-002']` (or comma-separated if required by legacy constraints).
3. **UI Display**: Use a `useMemo` filter to resolve IDs into objects for rendering:
   ```javascript
   const selectedObjects = coursesData.filter(c => subjects.includes(c.course_id));
   return selectedObjects.map(c => <span key={c.course_id}>{c.name}</span>);
   ```
